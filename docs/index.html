Overview

Infrastructure first, evidence second

AutoResearch is a shared-state orchestration system for ML experiments on SLURM clusters. It proposes new runs, prevents duplicate work through file-backed claims, ingests structured training results exactly once, tracks a global best record, and optionally escalates promising runs into a second-stage promotion hook.

The central thesis is simple: research search quality should be measured against compute spent, not just raw model quality.

Motivation

Why build autonomous experiment infrastructure?

Manual hyperparameter tuning wastes scarce GPU time, hides duplicate work, and makes it hard to reason about search efficiency as a first-class system property. AutoResearch reframes this as a controlled loop: propose, claim, execute, evaluate, update shared state, and optionally promote.

The project aims to answer a systems question before a modeling question: can an autonomous loop safely coordinate experiment search, preserve evidence, and quantify discovery per unit of compute?

Architecture

System design

proposal_engine ↓ orchestrator ↓ SLURM runner ↓ train.py ↓ write_result.py ↓ shared state layer (results.jsonl / claimed.json / global_best.json / promoted.jsonl) ↓ promotion hook (Tinker)

Core components

  • Proposal engine: emits one concrete next experiment from objective, prior outcomes, and policy mode.
  • Orchestrator: fingerprints candidates, acquires claims, submits jobs, and advances the control loop.
  • SLURM runner: renders deterministic job scripts and ships training to the cluster.
  • Ingestion layer: parses terminal metrics, computes reward, appends result records, and updates the best register.
  • Promotion hook: validates whether a run should escalate to a larger-scale Tinker check.

Shared state contracts

  • results.jsonl stores append-only terminal experiment records.
  • claimed.json prevents duplicate execution across workers.
  • global_best.json records the best known run under lock.
  • promoted.jsonl stores promotion decisions and downstream outcomes.
  • proposal_events.jsonl captures generated, skipped, rejected, and submitted proposal lifecycle events.
Experiment Loop

How one experiment moves through the system

  1. The proposer reads the current objective, prior results, and shared best state.
  2. It emits a hypothesis, config diff, reasoning trace, and deterministic fingerprint.
  3. The orchestrator claims the fingerprint or skips it if already completed or in flight.
  4. A SLURM job runs training and emits a structured metrics block to stdout.
  5. write_result.py ingests the terminal record and computes reward as improvement per TFLOP.
  6. The shared best register is updated if the new run is strictly better.
  7. If promotion criteria are met, the Tinker scale-up hook is attempted and logged.
Reward & Gates

Evaluation logic

The core efficiency metric is compute-aware:

reward = (baseline_val_bpb - current_val_bpb) / total_tflops_consumed

This turns model improvement into a research-efficiency signal. The system then layers on explicit gates for result status, best-run registration, and promotion threshold checks.

Results

What the matrix actually showed

The matrix validated the infrastructure: isolated condition roots, proposal event logging, clean ingestion, summary generation, and both promotion-path branches all worked under the fixed harness.

It did not yet validate proposer superiority. Under the controlled 5-vs-5 budget, neither proposer arm produced a proposer-driven discovery beyond its own seed baseline. The history-aware arm finished with a slightly lower best validation score, but that best run was still the seed baseline for that condition.

Clean interpretation: v1 proves the orchestration and experiment-accounting pipeline. It does not yet prove that history-aware proposals outperform the baseline policy under this small search budget.
Visibility

What v1 added beyond a plain runner

  • History-aware proposal mode as a runtime flag, persisted with results and proposal events.
  • Reasoning traces captured in append-only JSONL for post-hoc inspection.
  • Experiment archaeology to visualize search trajectory and best-run changes.
  • Per-condition summaries for reproducible evidence and lightweight reporting.
  • Promotion verification in both normal and forced-threshold modes.
Takeaways

What v1 proves

  • Distributed experiment coordination works reliably without a database.
  • File-backed claims and shared best updates behave correctly under isolated systems tests.
  • Terminal training results ingest exactly once and produce stable summary artifacts.
  • Proposal telemetry is now rich enough to compare generated, submitted, and skipped work.
  • The promotion path is verified end-to-end, including explicit forced-path coverage.
Future Work

Where v2 should go

  • Run larger proposer budgets or broader search spaces so proposer quality can be evaluated more meaningfully.
  • Pin shared baselines and seeds across conditions to reduce cross-arm variance.
  • Feed promotion outcomes back into proposal generation instead of treating them as terminal side effects.
  • Improve Tinker connectivity so the forced promotion path ends in a real scale-up rather than a logged network failure.
